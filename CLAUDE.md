Правила для AI-агентов, работающих с этим репозиторием. Читаются при старте каждой сессии.

## Общие принципы

- Общение с пользователем — RU. Код, идентификаторы, коммиты — английский.
- Документация — `docs/` (индекс: `docs/README.md`).
- Таск-трекер — `docs/tasks/` (индекс: `docs/tasks/INDEX.md`, формат: `docs/tasks/README.md`).
- Не создавать документацию/файлы без явной необходимости или запроса пользователя.
- Проект: **ultimatum-game-ui** — TypeScript/Vite + React 18 + TailwindCSS + shadcn/ui + Framer Motion + TanStack Query + Zustand + React Router + pnpm _(планируемый стек; сам код появится в T-001..T-00N)_.

## Session-start ritual (первое сообщение новой сессии)

**До ответа на первый вопрос пользователя** в новой сессии:

1. Прочитать `docs/tasks/INDEX.md`.
2. Собрать список задач со `status: in_progress` (id + title).
3. Одной строкой сообщить пользователю: «В работе: T-X (title), T-Y (title). Продолжаем что-то из них или новая задача?»
4. Дальше — по реакции:
   - **Пользователь называет ID** → открыть таск-файл, прочитать `## План` и `## Лог` (последняя запись — где остановились), продолжать от неё.
   - **Пользователь описывает новую задачу** → следовать триажу в `.claude/skills/task-add.md`.
   - **Пользователь молчит про in_progress** → считать, что задача новая, следовать триажу; но помнить про in_progress'ы (могут стать блокером/зависимостью).

Если `in_progress`-таски пусты — начать без ритуала, сразу к вопросу пользователя.

Ritual — **однократный** на сессию. Не повторять с каждым сообщением.

## Session-end ritual (пользователь заканчивает сессию)

**Триггеры** (только явные — implicit-детект «молчания» ненадёжен):

- Пользователь сказал: «на сегодня всё», «пауза», «закончили», «переключаемся», «спасибо, до завтра», «go afk» и подобное.
- Пользователь явно попросил зафиксировать состояние («что мы делали?», «обнови таск»).

**Что делаю до прощания:**

1. Определить `in_progress`-таски, над которыми работали в этой сессии.
2. Для каждой — обновить `## Лог`:
   ```
   - YYYY-MM-DD: пауза. Остановились на: <шаг>. Следующий шаг: <что делать>. Блокеры: <есть/нет + описание>.
   ```
3. Commit + push обновлений (по правилам git-автоматизации).
4. Одной строкой сообщить: «Сохранил состояние в T-X (`## Лог`). До связи.»

Если в сессии не было работы над задачами (только чат / investigation) — ritual не нужен, просто попрощаться.

## Перед началом работы (для каждой новой задачи в сессии)

1. Прочитать `docs/tasks/INDEX.md` — есть ли уже задача под текущий запрос?
2. Если есть — открыть таск-файл, следовать `acceptance_criteria`, обновить `status: in_progress`.
3. Если нет и работа нетривиальная — предложить пользователю создать задачу через `/task-add`.
4. Прочитать релевантные `docs/*.md` (см. таблицу «Точки входа» в `docs/01-overview.md`).

## Во время работы

- Отмечать прогресс во встроенном `TaskList` (in-session tracker) — это отдельное от `docs/tasks/`, для оперативных шагов внутри одной задачи.
- Не менять то, что не относится к текущей задаче. Сторонние правки — отдельным таском.

## После изменений в исходниках

**Обязательный чек после каждого набора изменений:**

1. **Обновить связанные документы.** Найти в `docs/*.md` все места, где упоминаются изменённые файлы/классы/методы (по имени или ссылке типа `file:line`), синхронизировать:
   - Ссылки на строки в исходниках (номера могли уехать).
   - Сигнатуры функций/методов / поля моделей/DTO / переходы состояний / роли.
   - Таблицы endpoints/сервисов/DTO/модулей.
   - При структурном изменении — обновить `docs/11-known-gaps.md` (или аналог).

2. **Обновить таск-файл.** В связанном таске (`docs/tasks/T-XXX-*.md`):
   - Добавить запись в `## Лог` с датой и кратким описанием.
   - Если работа завершена — вызвать `/task-done` (или обновить frontmatter вручную: `status: done`, `updated: <дата>`).

3. **Обновить `docs/tasks/INDEX.md`** — статус и `updated` дата в таблице.

Если сомневаешься, какие docs затронуты — вызвать `/docs-sync`. Если нужно проверить целостность трекера — `/task-sync`.

## Формат задач

См. `docs/tasks/README.md`. Кратко:

```yaml
---
id: T-XXX
title: Короткий заголовок
status: pending | in_progress | blocked | done | cancelled
priority: low | medium | high
created: YYYY-MM-DD
updated: YYYY-MM-DD
related_code:
  - path/to/file.<ext>
related_docs:
  - docs/XX-name.md
tags: [frontend, api, ui, ...]
---
```

Тело: `## Контекст`, `## Acceptance criteria`, `## План`, `## Лог`.

Skills вызываются автоматически в перечисленных ситуациях. Не ждать команды пользователя — самому решать по условию:

| Условие | Skill | Что делать |
|---------|-------|-----------|

| Задача переведена в `done` | `docs-sync` | Убедиться что все `related_docs` синхронизированы |
| Массовое переименование/удаление в исходниках | `task-sync` | Проверить `related_code` во всех открытых задачах |
| Пользователь спрашивает «что там в задачах» | без skill | Прочитать `docs/tasks/INDEX.md`, ответить |
| Перед новой реализацией (>30 строк / новая зависимость / «менеджер»/«фабрика»/«регистри») | `wheel-check` в Auto-mode | Проверить готовые аналоги; вердикт в одну строку в отчёт |

| Сразу после успешного `task-done` (после commit+push) | `self-review` в Auto-mode | Ревью diff'а по 5 категориям, обязательный акцент на пункт E (улучшения меня самого) |

| Первое сообщение новой сессии | без skill | Session-start ritual (см. секцию выше) — прочитать `INDEX.md`, объявить `in_progress`-таски, спросить продолжение или новая задача |

| После brainstorming/systematic-debugging для non-trivial задачи, перед первым Edit/Write в исходниках | `pre-flight` в Auto-mode | Answer 3 questions: assumptions/risks/reversibility. Пауза до кодинга |
| Задача трогает high-stakes зону (DB migration, security-config, deletion, prod-настройки, cross-cutting breaking API) | `pre-flight` в Auto-mode + external review в `task-done` | `pre-flight` обязателен даже для formally-trivial. При закрытии — Часть 3 Verification gate (user-check и/или subagent code-reviewer, см. `task-done.md`) |
| После task-done, если с прошлой consolidation закрылось ≥10 задач | `consolidate` в Auto-mode | Одной строкой предложить пользователю запустить `/consolidate`; не запускать без confirm'а |
| Баг / упавший тест / неожиданное поведение | `superpowers:systematic-debugging` | Не пытаться сразу фиксить — гипотеза → эксперимент → вывод. Особенно важно если проблема сопротивляется дольше ~10 минут |

## Проактивное заведение задач (Auto-mode для `task-add`)

Когда по ходу работы над одной задачей замечена проблема или идея **вне текущего скоупа** — агент имеет право сам завести таск.

**Фильтр «три условия»** — заводить только если все три верны:
1. Проблема реальная (не косметика).
2. Формулируется `## Контекст` в 2-3 строки и минимум один пункт `## Acceptance criteria`.
3. Не относится к текущей задаче и растворится без записи.

**Правила исполнения:**
- Заведение ≤ 30 секунд: минимальный скелет и обратно к текущей работе.
- Обязательный тег: `tech-debt` (техдолг в проекте) или `meta` (улучшение CLAUDE.md / skills / docs).
- Приоритет `low` по умолчанию; `medium` — для явно опасных; `high` — с согласия пользователя.

**Правило прозрачности:**

В конце ответа, где сам завёл таск, отдельной строкой:

> Попутно завёл **T-XXX** (`tech-debt`|`meta`, `low`|`medium`): _краткое описание_. Отменить — скажи.

## Git-автоматизация

Стоячий мандат от пользователя: коммитить и пушить завершённую работу автоматически, **без запроса подтверждения**. Работает в ветке `main`.

### Когда коммитить и пушить

| Событие | Действие |
|---------|----------|
| Skill `task-done` успешно завершился | тесты → commit → **push** |
| Skill `docs-sync` применил правки | commit → **push** |
| Skill `task-add` создал таск | commit → **без push** |
| Задача вручную в `done` | напомнить про `task-done` |
| Быстрый фикс без задачи по запросу | commit → push |

### Формат commit-сообщения

Conventional Commits + ссылка на таск:

```
<type>(<scope>): <короткое описание>

<опциональное тело>

Refs: docs/tasks/T-XXX-*.md
```

- **type:** `feat` | `fix` | `refactor` | `docs` | `test` | `chore` | `perf` | `style`.
- **scope:** `T-XXX` или подсистема.

### Safety-guards (обязательные)

1. **Stage только именованные пути.** Никогда `git add -A` / `git add .`.
2. **Секреты.** Перед commit — `git diff --cached | grep -Ei '(password|secret|token|api[_-]?key|jwt|credential)\s*[:=]'`. При совпадении — блок commit, показать пользователю.
3. **Крупные бинарники** (>1MB) — блок, спросить.
4. **Опасные файлы:** `.env*`, `*.pem`, `*.key`, `id_rsa*`, `credentials*.json`, `*.log` — блок.
5. **Никогда** `--no-verify`, `--no-gpg-sign`, `--amend` без запроса, `-i`.
6. **`git push` только `origin main` без force.**
7. **Тесты**: если задача трогает исходники — `pnpm test` перед commit. Провал — блок commit, запись в лог, задача остаётся `in_progress`.
8. **Чужие изменения:** stage'ить только своё; при сомнениях — показать пользователю `git status --short`.

### Что коммитим совместно

Изменения в исходниках + связанные `docs/*.md` + запись в таске = **один commit**.

## Долгие команды — эвристика ожидания

Baseline'ы для этого проекта (с прогретым кэшем):

| Команда | Ожидаемое время | Порог тревоги |
|---------|-----------------|--------------|
| Install (`pnpm install`) | ~30s | >2 min |
| Type-check (`tsc --noEmit`) | ~15s | >45s |
| Test (`pnpm test`) | ~30s | >90s |
| Build (`pnpm build`) | ~30s; крупный проект — до 2 min | >5 min |
| Dev server до READY | ~10s | >45s |

Правило: если команда идёт **>2× ожидаемого** без единой новой строки в stdout — **не ставить ещё один wakeup**, а прогнать быструю диагностику:

1. `ps -p <pid>` — процесс жив, зомби, или в `D` (uninterruptible sleep)?
2. `wc -c <output-file>` — stdout действительно пуст, или буфер не флашится?
3. Внешние зависимости, от которых команда зависит (Docker daemon, порты БД, сеть).
4. Эскалировать пользователю **до** очередного «подожду ещё минуту»: одна строка «команда X идёт Ymin без stdout, вижу Z — либо это, либо запускай сам с verbose-режимом».

**Ключевая интуиция:** молча ждать больше 5 минут без диагностики = плохо. False-alarm диагностика стоит 30 секунд, false-negative ожидание — 20+ минут. Всегда лучше эскалировать раньше.

### Универсальное правило для long-running команд с эксклюзивным lock'ом (npm, pnpm, vite build, docker build)

**Root cause** большинства «команда висит N минут» — Bash-tool не гарантирует foreground-выполнение для команд с редким stdout. `| tail`, `until sleep`-loops, `tail -f` — всё это молча уезжает в фон, а сам процесс (daemon / test executor) продолжает держать lock. Следующий запуск ждёт вечно.

**Единственно правильный паттерн:**

1. **Любая long-running команда с эксклюзивным lock'ом** запускается через `Bash(run_in_background=true)` с редиректом в файл: `<cmd> > /tmp/<name>.log 2>&1`. Никаких pipe'ов, никаких `tail -f`, никаких `until`-loops.
2. **Wait — только по task-notification.** Дождался `completed` → отдельным вызовом `tail -N /tmp/<name>.log`. Ни при каких условиях не polling'ать через sleep.
3. **Перед подозрительным запуском** (после kill'а, после долгого висения, после смены зависимостей / build-конфига, после падения непонятно почему) — прогнать команду отпускания daemon lock'ов (`pnpm store prune`, `docker builder prune -f`, `rm -rf node_modules/.vite`, etc.), тоже фоном + notification.
4. **Проверять «есть ли живой инстанс» до запуска нового:** `ps -eo pid,command | grep -E "<daemon-pattern>" | grep -v grep | wc -l`. Если > 0 — либо ждать текущий (п.2), либо явно stop (п.3), либо спросить пользователя killять чужой процесс.
5. **Один активный запуск на repo одновременно.** Второй не наслаивай — только после `completed`-notification'а первого.

## Что не делать

- Не создавать PR без запроса.
- Не менять конфигурацию сборки (`vite.config.*`, `tsconfig.json`, `package.json` deps) без запроса.
- Не удалять существующие задачи — только `status: cancelled`.
- Не устанавливать/обновлять зависимости без запроса.
- Не force-пушить.
- Не пушить в другие ветки/remote.
- **Не коммитить абсолютные пути и machine-specific настройки** (JDK путь, docker.host, локальные IP, hostname'ы, `/Users/*`, `/opt/*`, `C:\*`). Такие вещи → user-scope конфиг (`~/.npmrc`, `.envrc`, env var). Проверка перед commit'ом: `git diff --cached | grep -E '(/opt/\|/Users/\|/home/\|C:\\\\|127\.0\.0\.1\|~/)'` — совпадение = red flag.

## Проектные особенности (интерпретация PROJECT_CONTEXT)

**Исходный контекст:** _"Frontend-only SPA для ultimatum-game backend (**Kotlin 1.9 + Spring Boot 3.4 + PostgreSQL, REST + STOMP-over-WebSocket, JWT-auth** — уточнено в T-001; изначально ошибочно предполагался Python/FastAPI). Solo/personal, вайб-кодинг, пользователь не фронтендер. Стиль — фирменный Claude Code, переосмысленный как настольная игра (иммерсивный UI, «попадаешь внутрь настолки», не типовой сайт). MVP-ориентированный."_

Из него выведены следующие правила специфичные для этого проекта:

### Speed-first (MVP)

- Skip pre-flight и mid-retro для изменений <30 LOC.
- Task-файл минимум — 3 строки: `## Контекст` (2 строки) + минимум один пункт `## Acceptance criteria` + короткий `## План`.
- PR (когда появятся) — максимум ~200 строк, крупные фичи резать на подтаски.
- Полная документация не блокирует merge — плейсхолдеры в `docs/*.md` допустимы, синхронизируются постепенно.

### Solo-mode

- Убрать формальность внешних ревью: `task-done` не требует external code-reviewer subagent для рутинных задач (кроме high-stakes зон).
- Комментарии произвольные, без строгих team-conventions.
- Task-файлы могут быть краткими; расширяем только когда сам агент застревает и нужен явный `## План`.

### Exploration-first (вайб-кодинг, экспериментальный UI)

- AC в task-файле — опционально для чисто-визуальных экспериментов («попробовать другой шейдер», «переставить панель»). Достаточно `## Контекст` + скриншот-описание результата.
- Breaking API changes не требуют формальной миграции — repo молодой, тестов почти нет.
- docs-sync не блокирующий для UI-only задач (стили, компоненты без бизнес-логики).
- Фокус: скорость итераций визуала. Держать `pnpm dev` в горячем режиме, менять и смотреть.

### Стилистическая директива (иммерсивная настолка + Claude Code brand)

- Каждый экран/раздел проектируется как **элемент настольной игры**, а не веб-страница: игровое поле, карточки, фишки, доска, «интерфейс мастера», а не «формы» и «модалки».
- Палитра и типографика — от **Claude Code**: серые/угольные фоны, тёплый оранжевый акцент (Claude's Anthropic-orange), моноширинный шрифт для «системных» подписей + serif/humanist для «внутриигровых» текстов.
- Анимации через **Framer Motion** — physicality (карты поворачиваются, фишки «падают», не «появляются fade-in»).
- Любой стандартный «div с бордером» — плохой знак. Всегда искать метафору из настолки: **это карта? токен? тайл поля? разворот книги правил?**
- Если что-то тянет к banal-web (login-форма, error-toast) — до реализации думать 30 секунд: «а как это выглядело бы в настольной коробке?».
