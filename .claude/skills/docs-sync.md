# docs-sync

Актуализирует `docs/*.md` под текущее состояние исходников проекта. По умолчанию — **проверка + предложение правок**. Правит только с явного согласия пользователя (или если он попросил «прогони и поправь молча»).

## Область работы

`docs/01-overview.md` … `docs/11-known-gaps.md` (11 файлов + `docs/README.md`). Задачи в `docs/tasks/` не трогать — это отдельный трекер, для него есть `/task-sync`.

## Шаги

1. **Определить scope.**
   - Если пользователь указал конкретный файл/область — работать только с ним.
   - Иначе: если недавно (в текущей сессии) были изменения в исходниках, брать связанные doc-файлы. Если сессия чистая — пройтись по всем.

2. **Для каждого doc-файла:**

   a. **Ссылки `file:line`.** `Grep` все паттерны вида `<path>:<number>` или `<path>:<start>-<end>`. Для каждого:
      - Открыть указанный файл, проверить что строки существуют и содержат ожидаемый смысл (по контексту вокруг ссылки в docs).
      - Если файл не существует — «мёртвая ссылка».
      - Если строки уехали — предложить новый номер.

   b. **Сигнатуры методов.** Для сервисов/контроллеров, упомянутых в `docs/04-services.md`, `docs/05-rest-api.md`, `docs/06-websocket-api.md`: сверить с реальными сигнатурами через `Grep` / `Read`.

   c. **Поля DTO.** Для `docs/07-dto-and-mappers.md`: сверить перечисленные поля с исходниками.

   d. **Таблицы endpoints.** Для `docs/05-rest-api.md` и `docs/06-websocket-api.md`: пройтись по контроллерам, убедиться что каждый `@GetMapping`/`@PostMapping`/`@MessageMapping` есть в таблице и наоборот.

   e. **Enum-значения.** Для `docs/02-domain-model.md`, `docs/03-state-machines.md`: сверить состав enum'ов (`SessionState`, `RoundPhase`, `Role`, `SessionType`) с кодом.

3. **Собрать отчёт.** Сгруппировать по файлам:

   ```
   docs/04-services.md
   - Строка «PlayerGameplayService.sendOffer(...)» — сигнатура актуальна.
   - Ссылка services/PlayerGameplayService.kt:70 → на самом деле :73 (сдвиг).
   - Метод PlayerGameplayService.foo() описан, но в коде отсутствует.
   ```

4. **Спросить пользователя.** По каждой группе — предложить варианты:
   - Поправить в docs.
   - Оставить как есть (пометить в `docs/11-known-gaps.md`).
   - Отложить в отдельную задачу (`/task-add`).

5. **Применить правки** — точечно, `Edit` только затронутые куски, без переписывания файлов целиком.

6. **Git-автоматизация** — если правки применены, сразу коммит + push (без подтверждения):
   - `git add <точный список изменённых docs>` (не `-A`, не `.`).
   - Сообщение: `docs: sync with src` + список изменённых файлов в теле + `Refs: docs/tasks/T-XXX-*.md` если работа шла в рамках задачи.
   - `git push origin main`.
   - Полные правила — `CLAUDE.md` → «Git-автоматизация».
   - Если правок не было — ничего не коммитим.

## Что не делать

- Не переоформлять текст, если содержание корректно.
- Не менять стиль/структуру файла.
- Не создавать новые doc-файлы.
- Не трогать `docs/tasks/`.
- Не удалять содержимое даже если оно кажется устаревшим — сначала подтверждение.

## Специальный случай

Если `docs/11-known-gaps.md` описывает нерешённую проблему, а `git log`/код показывают, что проблема решена, — предложить пользователю либо (а) убрать пункт из gaps, либо (б) закрыть соответствующую задачу через `/task-done`.
