# task-done

Закрыть задачу: обновить frontmatter, лог и INDEX.

## Аргументы

- **Обязательный:** ID задачи (`T-XXX`).
- **Опционально:** одна фраза для записи в лог.

Если пользователь не указал ID — попросить одной фразой.

## Verification gate

**Перед шагом 4 (flip'ом status → done)** — обязательный проход `superpowers:verification-before-completion`.

### Часть 1. Технический verification

- Не заявлять что задача готова / passing / done без запуска verification-команд и без проверки их выхлопа.
- Если задача трогала исходники — прогнать релевантную build/test-команду из `CLAUDE.md` (см. `{{TEST_COMMAND}}` / `{{BUILD_COMMAND}}`) и посмотреть **зелёный ли** — не «наверное прошло», а буквально exit code 0 + маркер success в stdout.
- Если задача трогала `docs/**` — проверить что все ссылки актуальны и все упомянутые файлы действительно существуют.
- Если задача трогала CLAUDE.md/skills — проверить что frontmatter корректный (name, description) и что маркеры триггеров непротиворечивы.

### Часть 2. AC-deviation check (обязательный)

Прочитать `## Acceptance criteria` таск-файла **пункт за пунктом**. Для каждого — честно ответить:

- **Реализовано буквально** — сделано ровно то, что описано.
- **Отклонение** — пункт пропущен, заменён на другой подход, интерпретирован иначе, чем в тексте AC.

Если есть хоть одно отклонение — **до commit'а** одной строкой уведомить пользователя:

```
AC-отклонения при закрытии T-XXX:
- Пункт N: AC говорит [X], я сделал [Y]. Причина: [коротко].
- Пункт M: AC говорит [Z], пропущено. Причина: [коротко].
Ок закрывать так, или переделать?
```

Дождаться реакции:
- **Пользователь одобрил** → зафиксировать отклонения в `## Лог` таска, продолжать шаг 4.
- **Пользователь возразил** → `status` остаётся `in_progress`, вернуться к работе.

**Не оправдываться в `## Лог` постфактум.** Уведомление ДО commit'а — не после.

### Часть 3. External review (только для high-stakes)

**Триггер:** задача трогает high-stakes зону — DB миграции, security-конфиги, deletion (файлов/данных), prod-настройки, cross-cutting breaking API changes. Определение зон — то же, что для `pre-flight` (см. CLAUDE.md → Проактивные триггеры).

Для не-high-stakes задач — пропускаем эту часть.

**Механизм по типу high-stakes:**

| Тип | Механизм review |
|-----|-----------------|
| Security-config, deletion, prod-настройки | **Explicit user-check.** Показать пользователю `git diff --cached` (или summary если diff большой), явно спросить: «Это high-stakes изменение. Проверил ли ты? Ок закрывать? (y/n)». Ждать `y`. |
| DB миграции | **User-check + subagent code-reviewer.** Диспатчим subagent с prompt «review this DB migration for safety» (см. `superpowers:requesting-code-review` для формулировки), параллельно user'у. Оба должны дать «ok». |
| Cross-cutting breaking API changes | **Subagent code-reviewer** (эксплуатационно чистая проверка impact analysis + consumers). Fallback на user'а если subagent недоступен. |

**Что делать при возражении:**

- Пользователь / subagent сказал «нет» / выявил проблему → `status` остаётся `in_progress`, зафиксировать в `## Лог` конкретное возражение. Задача не закрывается.
- Пользователь молчит >5 минут (не подтвердил и не возразил) → **не** предполагать «ok»; переспросить один раз, затем оставить задачу в `in_progress` и завершить сессию через `session-end ritual`.

### Что делать при провале verification (любой части):

- `status` НЕ меняется на `done`.
- В `## Лог` — запись «verification блокировала закрытие: <что именно не сошлось>».
- Сообщить пользователю. Задача остаётся `in_progress`.

Только verification-passed (все три части) — переходим к шагу 4.

## Шаги

1. **Открыть файл** `docs/tasks/T-XXX-*.md` через `Glob`. Если нет — сказать пользователю и остановиться.

2. **Проверить acceptance criteria.** Прочитать секцию `## Acceptance criteria`:
   - Если есть незакрытые `- [ ]` — предупредить пользователя, спросить: всё равно закрыть или сначала закрыть чекбоксы?
   - Не пытаться самостоятельно проставить `- [x]`; это ответственность автора работы.

3. **Проверить docs.** По `related_docs` в frontmatter — были ли соответствующие файлы обновлены (по git-статусу или по здравому смыслу для контекста). Если сомнения — предложить пользователю сначала запустить `/docs-sync`.

4. **Обновить frontmatter:**
   - `status: done`
   - `updated: YYYY-MM-DD` (сегодня)

5. **Добавить запись в `## Лог`:** `- YYYY-MM-DD: закрыта. <фраза пользователя, если была>`.

6. **Обновить INDEX.md:**
   - Убрать строку из «Открытые задачи».
   - Добавить в «Закрытые задачи» с колонками `ID | Название | Статус | Закрыто | Файл`.

7. **Git-автоматизация** (см. `CLAUDE.md` → раздел «Git-автоматизация»):
   - Если задача трогала исходники — запустить команду тестов из `CLAUDE.md` (`{{TEST_COMMAND}}`; примеры: `./gradlew test`, `npm test`, `cargo test`, `pytest`, `go test ./...`). При падении: **не коммитить**, `status` остаётся `in_progress`, в `## Лог` записать «commit заблокирован: X тестов упали», сообщить пользователю. Пункты 4-6 откатить.
   - Тесты прошли (или не запускались) — commit + push связанных файлов.
   - Формат сообщения: см. Git-guide в CLAUDE.md.

8. **Отчёт пользователю:** одна строка — «T-XXX закрыта, запушено (`<hash>`)», плюс предупреждения если были.

## Ограничения

- Не удалять таск-файл.
- Не менять содержимое `## Контекст`, `## Acceptance criteria`, `## План`.
- Если пользователь ошибочно закрыл — можно откатить обратным ходом (`status: in_progress`, запись в лог).
