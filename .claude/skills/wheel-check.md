# wheel-check

Прогон 3–5-минутной проверки перед новой реализацией: «а не изобретаю ли я велосипед».

Смысл: чаще всего то, что кажется «надо написать самому», уже есть — либо в проекте, либо в фреймворке, либо в одной из уже подключённых зависимостей. Стоимость проверки — минуты, стоимость велосипеда — часы + вечный техдолг.

## Аргументы

- **Explicit:** `/wheel-check <короткое описание того, что собираюсь реализовать>`.
- **Auto-mode:** без аргументов, скилл сам знает контекст текущего шага.

## Шаги

1. **Сформулировать в одну фразу**, что именно я собираюсь реализовать. Написать пользователю (не более 15 слов).

2. **Проверить проект:**
   - `Grep` по исходникам на 2–3 ключевых слова из формулировки.
   - Прочитать релевантный `docs/*.md` — есть ли уже сервис/модуль с похожей ролью.

3. **Проверить фреймворк / уже подключённые зависимости:**
   - Смотрю манифест зависимостей (`package.json` / `build.gradle*` / `Cargo.toml` / `pyproject.toml` / `go.mod` / `pom.xml` / etc.) — какие библиотеки уже есть.
   - Соотнести мою задачу с типовыми механизмами фреймворка: логи, MDC/tracing, health/metrics, auth-фильтры, кеш, сериализация, валидация, миграции, DI, планировщики. Если моя задача накрывается одним из них — велосипед не строим.

4. **Проверить идиомы языка:**
   - Есть ли встроенная примитива для того, что я задумал (thread-local, scope-based cleanup, data-класс, sealed-тип, монада-обёртка, comprehension, etc.)?
   - Не писать «менеджер» / «регистри» / «фабрику» без сильной причины — это часто и есть велосипед.

5. **Docs pre-check** — что уже задокументировано про эту область:
   - Прочитать `docs/*.md` для затронутого модуля/слоя. Обратить внимание на:
     - Явно зафиксированные инварианты («роль X не может быть Y», «поле Z обязательно»).
     - Уже задокументированные ограничения / компромиссы («мы намеренно не поддерживаем W»).
     - Правила именования / формата, которые я собираюсь нарушить.
   - Если моё планируемое изменение противоречит уже задокументированному инварианту — **остановиться и уведомить пользователя**, а не «незаметно поменять поведение».

6. **Impact analysis** — кто зависит от того, что я собираюсь трогать:
   - `Grep` по каллерам изменяемого класса/функции/поля.
   - Проверить: сколько мест в исходниках реально используют это? Есть ли внешние потребители (открытый API, отдаваемый по контракту в OpenAPI/AsyncAPI/proto/schema)?
   - Если >5 каллеров или есть публичный API-контракт — это уже не «локальное изменение», это breaking. **Уведомить пользователя** в отчёте.

7. **Вывести вердикт** — 3 варианта в отчёте пользователю:
   - **«Пиши сам».** Обосновать: почему готовое не подходит, что именно уникально.
   - **«Используй X».** Указать конкретную либу/класс/паттерн, коротко чем заменяет мою идею.
   - **«Гибрид».** X делает Y%, остаток — своими руками. Уточнить что именно.

8. **Отчёт пользователю** — 3–7 строк. Один из трёх вердиктов + короткое обоснование + строка «Docs: [упомянуть если нашёл инвариант]» + строка «Impact: N каллеров / API-контракт [да/нет]». Не расписывать шаги «как я искал» — сразу к выводу.

9. **Только после отчёта — писать код.** Если вердикт «используй X» — соответственно править план.

## Auto-mode: агент запускает wheel-check сам

Скилл вызывается **до** написания кода, если срабатывает хотя бы одно из условий:

- Собираюсь добавить новую зависимость (любую) в манифест.
- Собираюсь создать новый файл длиннее ~30 строк.
- Собираюсь написать «менеджер», «регистри», «фабрику», «диспетчер», «пул», «кэш», «валидатор», «конвертер» — любое существительное с суффиксом «-er/-or» / «-Registry» / «-Factory» / «-Manager». Классический маркер велосипеда.
- Собираюсь реализовать что-то, что типично покрывается фреймворком (MDC/tracing, health-check, метрики, sec-фильтр, кеширование, JSON-сериализация, валидация, i18n, миграции, DI, планировщик).

Если срабатывает — молча провести проверку (шаги 1–5) и в отчёте пользователю указать одной строкой: «Прогнал wheel-check: [вердикт]». Дальше действовать по вердикту.

## Что НЕ является поводом для wheel-check

- Однострочные фиксы, правки логов, добавление поля в DTO/структуру, изменение сообщения ошибки.
- Правки тестов.
- Правки `docs/*.md`.
- Rename переменной / файла.
- Отдельный маленький util-хелпер (<10 строк).

## Ограничения

- Проверка занимает максимум 3–5 минут. Если ушёл дольше — либо задача сложнее и стоит проговорить с пользователем, либо я слишком глубоко копаю: закрыть проверку с ближайшим найденным вариантом.
- Не переоценивать чужие либы. Если готовое требует 5+ конфигов и 2+ адаптера — «пиши сам» может быть правильным ответом. Указать это в отчёте.
- Не подменять `wheel-check` полноценным дизайном. Это gate «не велосипед», а не «правильная архитектура».
