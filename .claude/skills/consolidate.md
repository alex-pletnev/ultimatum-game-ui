# consolidate

Периодический sweep закрытых задач на повторяющиеся meta-наблюдения. Не заменяет per-task `self-review` — дополняет его: `self-review` ловит наблюдения в одной задаче, `consolidate` — паттерн через 10 задач.

Смысл: одна нота в категории E — интересно; та же нота в двух подряд — уже сигнал (правило T-025); та же нота в 3-4 из последних 10 — точно правило, а не совпадение.

## Аргументы

- **Explicit:** `/consolidate` (без аргументов — берёт последние 10 закрытых) или `/consolidate <N>` (берёт последние N).
- **Auto-mode:** без аргументов, срабатывает по условию — с последнего запуска (записан в `.claude/consolidation-state.json`) закрылось ≥10 задач.

## Шаги

1. **Determine window.** Читаю `docs/tasks/INDEX.md`, беру последние N закрытых задач (по дате в колонке `Закрыто`, по умолчанию N=10). Если задач меньше N — брать все закрытые, но информировать пользователя.

2. **Collect observations.** Для каждой задачи в окне:
   - Открыть таск-файл, найти `## Лог`.
   - Извлечь текст, относящийся к self-review (записи «self-review», «категория E», «замечено», «наблюдение», «мельком заметил» и подобные).
   - Также посмотреть на **тэги** (`meta`/`tech-debt`) — категория задачи полезна для группировки.
   - Собрать в plaintext'овый корпус.

3. **Group by theme.** Пройтись глазами (без ML — просто здравый смысл) и сгруппировать наблюдения по темам:
   - «Не запустил verification до commit'а» — сколько раз встречалось?
   - «Расширил скоуп задачи» — сколько раз?
   - «Пропустил pre-flight» — сколько раз?
   - Собственные наблюдения агента (E-category) — искать буквальные повторения слов и структур.

4. **Detect patterns.** Тема считается **паттерном**, если она встретилась в **2+ задачах** в окне. Если один-два тегмита — не паттерн, шум.

5. **Report.** Отчёт пользователю — компактный:
   ```
   ## Consolidation (последние N закрытых задач)

   Обнаружены паттерны:

   1. [Тема Х] — встречалась в T-A, T-B, T-C (3 задач). Предлагаемое действие: <заведи T-YYY meta / уточни правило Z в CLAUDE.md / поменяй skill Q>.
   2. [Тема Y] — T-D, T-E (2 задач). Предлагаемое действие: ...

   Одиночные наблюдения (не паттерны, но упомянуть):
   - <one-liner>: T-F.

   Что заводить как задачу? y/скажи какие.
   ```

6. **Wait for user.** Пользователь решает:
   - «Заводи всё» → для каждого паттерна вызвать `task-add` в Auto-mode с тегом `meta`, приоритетом `medium` (у паттернов приоритет выше single-task наблюдений).
   - «Только 1, 3» → завести выборочно.
   - «Ничего» → просто зафиксировать выполнение consolidation в state (см. шаг 7).

7. **Update state.** Записать `.claude/consolidation-state.json`:
   ```json
   {
     "last_run_date": "YYYY-MM-DD",
     "last_run_after_task": "T-XXX",
     "closed_tasks_count_at_last_run": N
   }
   ```
   Файл коммитится в git.

8. **Commit + push.** Если завелись task-add'ы — они commit'ятся по правилам task-add. `consolidation-state.json` — отдельный commit `chore(consolidate): sweep after T-XXX`.

## Auto-mode: агент вызывает consolidate сам

Условие: количество closed-задач в `INDEX.md` минус `closed_tasks_count_at_last_run` из state ≥ 10.

- Если условие срабатывает при `task-done` (после успешного закрытия таска) — **не запускать consolidation сразу**. Вместо этого одной строкой сообщить пользователю: «С прошлой consolidation закрылось X задач (порог 10). Запустить `/consolidate`?».
- Пользователь `y` → запуск. `n` / молчание → пропуск до следующего task-done, где условие снова проверится.

Не запускать чаще одного раза в 10 задач. Не запускать если window слишком мал (<3 закрытых задач за последний период — просто нечего анализировать).

## Что НЕ является поводом

- Проект новый (<10 закрытых задач за всю историю).
- Последний run был <5 задач назад.
- Пользователь только что явно отклонил вопрос про consolidation.

## Ограничения

- **Только читает — ничего не меняет автоматически.** Все action item'ы — через `task-add` с явным подтверждением пользователя.
- **Не анализировать closed задачи с `cancelled` статусом** — они не отражают реальный опыт.
- **Не углубляться в анализ.** Skill — sweeper, а не глубинный retrospective. Один прогон = максимум 5-10 минут работы. Если анализ занял больше — сузить window.
- **Не предлагать паттерн, если у него уже есть открытая задача.** Проверить `INDEX.md` на похожие открытые.
