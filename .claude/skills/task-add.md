# task-add

Создать новый таск-файл в `docs/tasks/` и добавить строку в `docs/tasks/INDEX.md`.

## Аргументы

- **Обязательный:** название задачи (короткое, на русском или английском).
- **Опционально:** приоритет (`low`/`medium`/`high`, default `medium`), теги.

Если пользователь не указал название — спросить одной фразой, не подсказывать варианты.

## Триаж перед созданием таска

**Перед шагом 1** — определить **тип задачи**. От типа зависит, какой процесс подключается дальше.

| Тип | Признаки | Процесс перед созданием таска |
|-----|----------|-------------------------------|
| **Trivial fix** | однострочный фикс, rename, добавление флага, правка сообщения ошибки, комментария; точечная правка `docs/*.md` | Ничего. Идти сразу к шагу 1. |
| **Feature / design work** | новое поведение, новый модуль/endpoint/DTO, изменение архитектуры; >1 файла с развилками дизайна | `superpowers:brainstorming` → spec-файл в `docs/superpowers/specs/`. Ссылка в `related_docs` таска. |
| **Bug / regression** | что-то работает не так, тест падает, поведение неожиданное | **НЕ brainstorming** (это не creative work). `superpowers:systematic-debugging`: гипотеза → эксперимент → root cause. Только после понимания причины — создание таска с чётким AC (repro + fix). |
| **Refactor / tech-debt** | изменение структуры без изменения поведения; поведение уже покрыто существующими тестами | Ничего до создания таска. Внутри задачи — `wheel-check` (не изобретаю ли уже готовое) + verification что tests не сломались. |
| **Investigation / research** | «посмотри как работает X», «есть ли Y», «оцени возможность Z» | Ничего до создания таска. Результат — задача типа `research`, closes с `## Лог` с выводами, не требует code changes. |
| **Emergency hotfix** | прод горит, критичный баг | Skip тяжёлые гейты (brainstorming, writing-plans, полное verification). Repro + минимальный fix + smoke-verify. Full retro и follow-up'ы — постфактум. |

**Как определяется тип:** сначала — по формулировке пользователя. Если неясно — одной фразой уточнить: «Это баг, фича или что-то другое?». Не выдумывать за пользователя.

Если пользователь явно сказал «просто заведи таск, не проектируй» — skip любые gate'ы, идти сразу к шагу 1.

## Шаги

1. **Проверить дубликат.** `Grep` по `docs/tasks/INDEX.md` — нет ли уже задачи с похожим названием. Если есть — показать пользователю и уточнить: продолжить/отменить.

2. **Определить следующий ID.** `Grep` по `docs/tasks/` паттерн `^id: T-` в frontmatter'ах или взять максимальный номер из имён файлов `T-XXX-*.md`. Новый ID = max + 1, 3 цифры (`T-004`, `T-042`).

3. **Собрать метаданные:**
   - `title` — из ввода пользователя.
   - `status: pending`.
   - `priority` — из ввода или `medium`.
   - `created`, `updated` — сегодняшняя дата (`YYYY-MM-DD`).
   - `related_code`, `related_docs`, `tags` — если пользователь не дал, оставить пустыми списками (`[]`), задача может дополнить их сама при работе.

4. **Написать файл** `docs/tasks/T-XXX-kebab-title.md` в формате из `docs/tasks/README.md`:
   - frontmatter из шага 3;
   - секции `## Контекст` (короткое пояснение), `## Acceptance criteria` (заготовка `- [ ] ...`), `## План` (пустой список), `## Лог` с записью `- YYYY-MM-DD: заведена.`

5. **Обновить INDEX.md.** Добавить строку в таблицу «Открытые задачи», сохраняя сортировку по ID (по возрастанию).

6. **Git-автоматизация:** commit создания карточки — но **без push**. Push произойдёт со следующим `task-done` или `docs-sync`, чтобы не пушить на каждую заведённую задачу.
   - `git add docs/tasks/T-XXX-*.md docs/tasks/INDEX.md`
   - Сообщение: `chore(tasks): add T-XXX <короткий title>`
   - Полные правила — `CLAUDE.md` → «Git-автоматизация».

7. **Отчёт пользователю:** ID, имя файла, ссылка. Не расписывать содержимое.

## Ограничения

- Не заполнять `Acceptance criteria` и `План` содержательно, если пользователь не дал деталей — оставить как placeholder. Задача — быстро завести карточку, а не проектировать за пользователя.
- Не менять существующие таски.
- Если название на английском, kebab-case из него; если на русском — кратко транслитерировать или взять ключевые слова на английском.

## Auto-mode: агент заводит задачу сам

Скилл вызывается **не только по явной команде пользователя**. Агент имеет право сам инициировать `task-add`, если по ходу работы над другой задачей заметил проблему или идею вне текущего скоупа.

### Фильтр «три условия»

Заводить таск в Auto-mode только если **все три** верны:

1. Проблема реальная — сломанное поведение, скрытый риск, дублирование, нарушение инварианта. Не косметика вроде «здесь можно чуть красивее».
2. Можно сформулировать `## Контекст` в 2-3 строках и минимум один пункт `## Acceptance criteria`.
3. Проблема не относится к текущей задаче и растворится без записи.

Если хоть одно условие не выполнено — **не заводить задачу**. Вместо этого одной строкой упомянуть в конце ответа: «Мельком заметил X, заводить не стал».

### Обязательные атрибуты

- **Тег:** ровно один из `tech-debt` (техдолг в проекте) или `meta` (улучшение CLAUDE.md / skills / docs-структуры).
- **Приоритет:** `low` по умолчанию; `medium` — если проблема явно опасная (потеря данных, security); `high` — не устанавливать без согласия пользователя.
- **Запись в `## Лог`:** `- YYYY-MM-DD: заведена автоматически в ходе работы над T-YYY.` (если родительской задачи нет — «в ходе ad-hoc работы»).
- **`related_code` / `related_docs`:** заполнить хотя бы одним элементом, иначе `/task-sync` и `/docs-sync` не смогут работать.

### Правила исполнения

- Заведение ≤ 30 секунд: минимальный скелет и обратно к текущей работе. Никакого планирования/анализа найденной проблемы прямо сейчас.
- Не отвлекаться на реализацию — только карточка.
- Не заводить дубликаты: проверить `INDEX.md` на похожие открытые задачи (по ключевым словам).

### Правило прозрачности

В конце ответа, где сам завёл таск, добавить **отдельной строкой**:

```
Попутно завёл **T-XXX** (`tech-debt`|`meta`, `low`|`medium`): краткое описание. Отменить — скажи.
```

Если пользователь просит отменить (например, «отмени T-XXX»):
- открыть таск-файл, поменять `status: cancelled`,
- в `## Лог` дописать `- YYYY-MM-DD: отменена по запросу пользователя.`,
- переместить строку из «Открытые» в «Закрытые» в `INDEX.md`.

### Что НЕ является поводом для Auto-mode

- Стилистические замечания к коду (форматирование, порядок импортов).
- «Можно было бы использовать более идиоматичный API» без реальной проблемы.
- Мысли типа «вот тут может быть багу когда-нибудь» без конкретного сценария.
- Проблемы, которые уже описаны в `docs/11-known-gaps.md` (сначала свериться).
