# pre-flight

Короткий gate между «знаю что делать» и «начал кодить». Три вопроса. Написать пользователю ответы. Только после этого — первый Edit.

Смысл: brainstorming ловит **дизайн**, systematic-debugging ловит **причину**. Ни один из них не спрашивает: «а что я на веру принимаю, что может сломаться, и как это откатить». Именно этот пробел закрывает pre-flight.

## Аргументы

- **Explicit:** `/pre-flight`.
- **Auto-mode:** без аргументов, срабатывает по условиям.

## Шаги

Написать пользователю ответы на **три вопроса**. Не длинно — 2-4 строки на каждый. Формат:

```
## Pre-flight (T-XXX)

1. Assumptions — что принимаю на веру, не выясняя:
   - <утверждение>
   - <утверждение>

2. Risks — что может сломаться от этого изменения:
   - <риск> → как ловим (тест, review, monitoring)
   - <риск> → …

3. Reversibility — если сломается, как откатим:
   - <способ отката: `git revert`, миграция вспять, restore backup, hotfix N>
   - <ETA отката: минуты / часы / нужен человек с доступом>
```

После отчёта — **пауза**. Дальше два варианта:

- **Есть неясности / пользователь возразил** → зафиксировать в `## Лог` таска, вернуться к brainstorming или задать уточняющий вопрос пользователю. Не кодить.
- **Всё чисто, пользователь молчит / одобрил** → писать код.

## Auto-mode: агент вызывает pre-flight сам

Срабатывает при **всех трёх** условиях одновременно:

1. Задача — non-trivial (тип `feature`, `refactor`, `hotfix` из триажа `task-add.md`). Trivial fix / investigation / docs-only — не срабатывает.
2. Уже прошёл соответствующий design/analysis gate: `superpowers:brainstorming` для feature, `superpowers:systematic-debugging` для bug (в рамках hotfix), готовый spec/plan для refactor.
3. Собираюсь сделать первый Edit/Write в исходниках для этой задачи.

Триггер разовый на задачу — второй pre-flight в рамках той же задачи не нужен.

**Дополнительный триггер (обязательный вне зависимости от условий выше):** если задача трогает **high-stakes зоны** — DB миграции, security-конфиги, deletion (файлов/данных), prod-настройки, cross-cutting breaking API changes — pre-flight обязателен, даже если задача formally тривиальна.

## Что НЕ является поводом

- Trivial fix (rename, typo, комментарий).
- Investigation / research без code changes.
- Docs-only задачи.
- Второй заход в ту же задачу (если pre-flight уже был раз).

## Ограничения

- **Не превращать в мини-план.** Только три пункта, каждый 2-4 строки. Если начал расписывать 20 строк на пункт — это уже writing-plans / brainstorming, не сюда.
- **Assumptions** — не «я предполагаю что компилятор работает». Это должны быть **проверяемые** утверждения о поведении системы, которые я решил не проверять сейчас. Пример: «предполагаю, что все callers `X` уже обрабатывают null» (можно проверить grep'ом, но не буду — assumed).
- **Risks** — конкретные сценарии, не общие слова. «Может сломаться логин» — плохо. «Если пользователь отправит X до Y, состояние Z будет inconsistent» — хорошо.
- **Reversibility** — фактическая процедура, а не «откатим коммит». Что именно, кто может это сделать, сколько времени.
- Если по одному из пунктов «не знаю» — это сигнал остановиться и вернуться на design/analysis stage, а не проскакивать.
