## Ответы на предыдущие открытые вопросы (после T-001)

| Вопрос | Ответ из upstream |
|--------|-------------------|
| Мультиплеер vs hot-seat | **Мультиплеер.** Каждый игрок — свой JWT + WS-сессия. Hot-seat не поддерживается на уровне API, но теоретически возможен через несколько браузер-табов с разными login'ами |
| Аутентификация | **JWT** (access 15 мин + refresh 14 дн), `POST /auth/quick-register {nickname, role}` — без пароля |
| Сохранение партии | **Persistent** (PostgreSQL). Перезагрузка фронта не теряет партию — можно переподключиться и подтянуть state |

## Технические долги / риски (замечены в backend'е, влияют на фронт)

Backend работает, но есть ограничения — их закрывать не мы, но фронт должен уметь жить с ними:

| Проблема | Влияние на фронт | Что делать |
|----------|------------------|------------|
| **Таймауты фазы не enforce'ятся** (`config.timeoutMoveSec` в схеме, но сервер не переключает фазу) | Раунд может «висеть» бесконечно | Локальный таймер + UI-подсказка admin'у: «раунд Nмин без движения → abort?» |
| **Refresh-token rotation отключена** (by design MVP) | Старый refresh работает до `exp` вечно | На фронте — 401-interceptor с попыткой `/auth/refresh` → retry, если снова 401 → logout |
| **Anti-impersonation на STOMP не проверяется** | Клиент теоретически может слушать чужой `/topic/.../player/{otherUserId}/offer` | Low-priority для solo-MVP; для prod — задача backend'а |
| **Порядок WS-сообщений между destination'ами не гарантирован** | `offerCreated` для offer A может прийти после `decisionMade` для offer B | Реагировать на текущий state (fetch `GET /current-round` при неоднозначности), не на порядок event'ов |
| **CORS для prod не готов** | Deploy заблокируется | Задача backend'а на prod-время |
| **Endpoint удаления сессии отсутствует** | Old sessions копятся в UI-списке | Клиентский фильтр (напр. `state != FINISHED` или `< 7 дней`) |
| **Двойное `POST /session/{id}/join`** — поведение не описано | Возможно idempotent, возможно 409 | Защитить UI-кнопкой (disable после успешного join) |

## Стилистические вопросы (будут решаться постепенно)

- **Что делать с `role: OBSERVER`?** Иммерсия «внутри настолки» странно ложится на observer'а — он не игрок за столом, а «зритель». Разный визуальный режим?
- **`FREE_FOR_ALL` vs `TEAM_BATTLE`** — разная физика (круглый стол vs две команды напротив). Разные лейауты или общий с адаптацией?
- **`RoundPhase = ABORTED`** — как это показать? «Ведущий отменил раунд» — что рисуется на «доске»?

## Security / performance concerns

- **JWT в `localStorage` vs `httpOnly cookie`** — по классике cookie надёжнее (XSS-immune), но `httpOnly` не читается из JS → нельзя вставить в STOMP CONNECT-фрейм. Backend, судя по всему, ожидает Bearer в заголовке → `localStorage` (или `sessionStorage`). Задача: минимизировать XSS-поверхность (никакой `dangerouslySetInnerHTML`, CSP).
- **WebSocket reconnect storm** при массовом обрыве сети — `@stomp/stompjs` имеет `reconnectDelay`, надо не делать exponential backoff кустарно.
- **Bundle size** при импорте `@stomp/stompjs` — проверить, что tree-shake работает.
